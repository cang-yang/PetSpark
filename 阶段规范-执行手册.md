# 智慧AI宠物管理平台 — 实训全流程阶段规范 V1.0

> 本文档是接下来所有开发工作的**唯一执行规范**。每一步都必须对照本文档执行，不得跳步。
> 核心原则：**一个功能一个分支一个 PR**；**需求驱动设计，设计驱动编码**；**每个阶段都有产出物，没有产出物视为该阶段未完成**。

---

## 0. 总体流程图（一句话版）

`需求分析 → 目标用户画像 + 用户故事 → 全网调研同类用户需求 → 功能优化与创新设计（基础+加分+特色）→ 技术设计（架构/数据库/接口）→ 逐功能开发（拉代码→建分支→开发→提交PR→合并）→ 测试 → 部署 → 答辩`

每个阶段有明确的**进入条件**、**执行动作**、**产出物**、**退出条件**。下一阶段必须满足上一阶段的退出条件才能开始。

---

## 阶段一：需求分析（Day 0-1）

### 进入条件
- 实训方案文档已读取并理解（已完成）。
- Git 仓库已初始化（远程 + 本地）。

### 执行动作
1. 从实训方案中提取**全部显性需求**（C端用户端、B端后台管理端的所有模块）。
2. 用表格形式列出每个模块的：功能名 / 使用角色 / 输入 / 输出 / 依赖。
3. 标注**基础功能（必做）**、**加分项（应做）**、**特色项（可做）** 三个优先级。
4. 识别**非功能需求**：安全（BCrypt/JWT/HTTPS/验证码/登录IP记录）、性能、可维护性、文档完整性。

### 产出物
- `docs/01-需求分析说明书.md` — 含需求清单表、优先级标注、非功能需求。

### 退出条件
- 需求清单覆盖方案中所有提及的模块。
- 每条需求都有唯一编号（如 REQ-C-001 用户注册）。
- 优先级标注完成。

---

## 阶段二：目标用户画像与用户故事（Day 1）

### 进入条件
- 阶段一产出物已完成。

### 执行动作
1. 识别**全部目标用户角色**（不限于方案中明示的，要主动扩展）：
   - C端：宠物主人（养宠新手/资深玩家）、潜在领养者、宠物服务消费者、社区爱好者。
   - B端：平台管理员、宠物服务商家、商品运营、内容运营。
   - 边缘角色：流浪动物救助者、宠物医生、寄养家庭。
2. 为每个角色建立**用户画像卡**：年龄/职业/痛点/目标/使用场景/技术熟练度。
3. 为每个角色编写**用户故事**，格式统一：`作为一个<角色>，我希望<动作>，以便<价值>`。
4. 每个用户故事编号（US-001），并关联到阶段一的需求编号（REQ-xx）。

### 产出物
- `docs/02-用户画像与用户故事.md` — 画像卡 + 用户故事清单 + 需求映射表。

### 退出条件
- 至少覆盖 5 个核心角色。
- 每个角色至少 5 条用户故事。
- 每条用户故事都能追溯到至少一个需求编号。

## 阶段三：全网调研与需求优化（Day 1-2）

### 进入条件
- 阶段二产出物已完成。

### 执行动作
1. **联网搜索同类平台**：波奇宠物、E宠、宠物医院SaaS、PetSmart、宠物领养平台等，调研它们的真实用户评论、痛点、好评点。
2. 调研**宠物行业用户未被满足的需求**：扫码识别品种、症状自查、AI喂养建议、社区提问、寄养匹配、寻宠启事等。
3. 调研**讯飞星火大模型能力边界**：能做什么（对话/推荐/生成/理解）、不能做什么、接入方式、成本。
4. 基于调研，对阶段一的需求进行**优化与扩充**：
   - 合并重复需求。
   - 升级模糊需求为明确需求。
   - 新增**加分项**（超出基础但有价值）：如 AI 健康档案分析、智能客服、个性化喂养方案。
   - 新增**特色项**（差异化亮点）：如宠物品种拍照识别、行为AI解读、社区UGC+AI摘要、领养匹配算法。
5. 输出**最终功能清单**（带优先级、用户故事关联、技术可行性初判）。

### 产出物
- `docs/03-调研报告与需求优化.md` — 调研来源链接 + 痛点归纳 + 优化后功能清单 + 加分项 + 特色项。

### 退出条件
- 至少调研 5 个同类产品/3 篇行业文章。
- 优化后清单包含至少 3 个加分项、2 个特色项。
- 每条都有技术可行性标注。

## 阶段四：技术设计（Day 2-3）

### 进入条件
- 阶段三产出物已完成并被全员确认。

### 执行动作
1. **架构设计**：前后端分离图、模块划分、技术栈最终确认（Spring Boot 3 / Vue2 / MyBatis-Plus / MySQL 8 / 讯飞星火 / ECharts）。
2. **数据库设计**：E-R图、表结构、字段、约束、索引、初始数据。所有表用前缀统一命名。
3. **接口设计**：RESTful 规范、统一返回结构、错误码规范、分页规范、鉴权规范。用表格列出每个接口：方法/路径/入参/出参/鉴权/对应需求。
4. **模块拆分**：将功能拆成可独立开发交付的**功能块**，每块对应一个 PR。这是后面"一功能一PR"的粒度依据。
5. **AI 接入设计**：星火大模型调用封装、提示词模板、限流、异常降级。

### 产出物
- `docs/04-概要设计说明书.md` — 架构 + 模块划分。
- `docs/05-数据库设计说明书.md` — E-R + 表结构。
- `docs/06-接口设计说明书.md` — 全部接口清单。
- `docs/07-PR拆分清单.md` — 每个PR的内容、依赖顺序、预估工时。

### 退出条件
- 所有文档完成并通过内部评审。
- PR 清单按依赖关系排好执行顺序。

## 阶段五：逐功能开发（Day 3-17）—— 核心阶段

### 进入条件
- 阶段四全部产出物完成。
- Git 远程仓库、main 分支、开发规范已就绪。

### 单个功能的标准工作流（每个PR都严格走这套）

```
1. git checkout main && git pull origin main      # 永远基于最新main
2. git checkout -b feature/<模块名>-<功能名>     # 命名规范
3. 编码（遵循Java编码规范，方法注释、异常处理、日志）
4. 本地自测（JUnit单测 + 手动验证）
5. git add / git commit（提交信息规范见下）
6. git push origin feature/<...>
7. 在远程创建Pull Request，填写PR模板
8. 至少1人Code Review，通过后合并到main
9. 合并后删除本地与远程分支
10. 下一个功能回到第1步
```

### 提交信息规范
- feat: 新功能
- fix: 修复
- docs: 文档
- refactor: 重构
- test: 测试
- chore: 杂项
- 格式：`feat(模块): 简述` ，正文写"做了什么/为什么/影响"。

### PR 模板（每个PR必填）
- 关联需求编号 / 用户故事编号
- 本次改动内容
- 自测情况
- 影响范围
- 截图或日志（如适用）

### PR 拆分顺序建议（由底向上，先骨架后血肉）
1. PR-01 项目脚手架（Spring Boot+Vue2+数据库连接）
2. PR-02 数据库初始化与基础数据
3. PR-03 用户注册+登录+JWT+拦截器
4. PR-04 用户信息管理
5. PR-05 宠物信息管理（CRUD）
6. PR-06 商品管理
7. PR-07 订单管理
8. PR-08 服务预约
9. PR-09 后台首页ECharts统计
10. PR-10 社区/消息通知
11. PR-11 图片上传
12. PR-12 收养/寻宠地图（特色）
13. PR-13 讯飞星火AI对话封装
14. PR-14 AI对话落地（宠物护理问答）
15. PR-15 AI智能推荐落地
16. PR-16 加分项：AI健康档案分析
17. PR-17 特色项：拍照识别品种 / 行为解读（任选）
18. PR-18 项目部署脚本与文档

### 退出条件
- 全部PR合并完成。
- main 分支可一键启动并跑通主流程。
- 无 P0/P1 未解决缺陷。

## 阶段六：测试（Day 17-18）

### 进入条件
- 阶段五全部PR合并。

### 执行动作
1. 单元测试覆盖率核查（核心业务方法必须有单测）。
2. 接口测试（Postman/自动脚本，覆盖全部接口的正常+异常路径）。
3. 端到端测试：注册→登录→浏览→下单→预约→AI对话→AI推荐 全链路。
4. 安全测试：未授权访问、越权、SQL注入、密码加密验证、Token过期。
5. 性能测试：首页接口响应时间、并发下单。
6. 缺陷登记与修复（每个修复走标准PR流程）。

### 产出物
- `docs/08-测试报告.md` — 用例 + 执行结果 + 缺陷清单 + 修复记录。

### 退出条件
- 全部用例执行完毕，无遗留严重缺陷。

## 横向规范（贯穿全程）

### 文档规范
- 所有文档放 `docs/` 目录，命名 `序号-名称.md`。
- 每个 PR 如涉及设计变更，必须同步更新对应文档。

### 分工规范
- 每个PR指定一名主负责人 + 一名Review人。
- 组长负责排期与阻塞协调。

### 每日规范
- 每日站会：昨天做了什么 / 今天做什么 / 是否阻塞。
- 每日提交日报（实训方案明确要求）。

### 质量红线
- 禁止直接 push main，必须走PR。
- 禁止合并未通过Review的PR。
- 禁止提交编译失败的代码。
- 禁止删除他人的文档产出。

### 创新驱动
- 阶段三的加分项与特色项，凡技术可行的，至少落地2个。
- 鼓励在任意阶段提出优化，但必须走"需求变更→设计更新→PR"流程，不得私自加功能。

## 流程速查（贴在屏幕边）

```
需求分析 → 用户故事 → 联网调研 → 优化清单
   → 架构设计 → 数据库设计 → 接口设计 → PR拆分
      → 拉main → 建分支 → 编码 → 自测 → 提PR → Review → 合并
         → 测试 → 部署 → 答辩
```

**一功能，一分支，一PR。绝不直推 main。**
